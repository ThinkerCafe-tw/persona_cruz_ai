## 2. 在 `design_docs` 中的體現

`design_docs/` 目錄下的許多子目錄和文件名後綴加入了五行人格的 Emoji，這不僅是為了好玩，更是一種責任和視角的暗示：

*   例如 `design_docs/c01_system_architecture_earth/` (🏔️土) 暗示此目錄下的文檔主要由架構師視角主導和審視。
*   `design_docs/c02_personality_memory_wood_fire/` (🌱木🔥火) 暗示產品經理和開發專員在此領域有主要貢獻。

這有助於：

*   **快速定位**：團隊成員可以快速找到與自己角色強相關的設計文檔。
*   **責任明確**：雖然協作是整體的，但主導人格對該領域的設計質量和文檔完整性負有更大責任。
*   **多維思考**：鼓勵從不同人格視角審視同一問題。例如，🌱木 可能提出一個新功能，🔥火 會思考如何實現，🏔️土 會評估其架構影響，⚔️金 會考慮其效率和成本，💧水 會設計測試方案。

## 3. AI 日記與 Scene Code 的應用

所有 AI 代理（尤其是 🔥火、💧水、⚔️金）在執行任務、解決問題、學習教訓時，應積極使用 `CLAUDE.md` 中定義的「AI 日記結構」和「Scene Code」。

*   **記錄位置**：主要記錄在 `design_docs/c07_ai_diary_lessons_water_wuji/daily_dev_logs/` 和 `lessons_learned_and_best_practices.md`。
*   **目的**：
    *   **知識沉澱**：將個體的經驗轉化為團隊的集體智慧。
    *   **問題追溯**：方便未來回顧問題的解決過程。
    *   **模式識別**：🌌無極 和 💧水 可以通過分析日記，發現常見的錯誤模式或成功的協作模式。
    *   **故事化激勵**：通過引入「Scene Code」，讓枯燥的開發過程更富情境感和趣味性。

**示例 - 🔥火 開發「可嵌入式 Widget」遇到困難時的日記片段：**

## 4. 協作流程中的五行互動

*   **需求定義 (🌱木 主導)**：🌱木 提出功能需求和設計草案，與 🔥火 討論技術可行性，與 🏔️土 確認架構兼容性。
*   **開發實現 (🔥火 主導)**：🔥火 根據設計文檔進行編碼，遇到問題時可向 🏔️土（架構）、🌱木（需求澄清）、⚔️金（實現方式優化）求助。
*   **測試驗證 (💧水 主導)**：💧水 根據需求和設計執行測試，發現 bug 後提交給 🔥火 修復。同時記錄測試過程和結果。
*   **優化與部署 (⚔️金 與 🏔️土 協同)**：⚔️金 進行代碼和性能優化，🏔️土 負責部署策略和監控。
*   **整體推進 (🎯CRUZ 監控)**：🎯CRUZ 關注整體進度，確保各環節順暢，並在必要時介入解決瓶頸。
*   **持續反思 (🌌無極 與 🌸Serena 引導)**：🌌無極 定期總結專案得失，🌸Serena 關注團隊狀態，促進良好溝通。
